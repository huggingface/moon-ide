# ADR 0017 — Neutralise project `restart:` policies via an override file

Date: 2026-05-28
Status: accepted; amends
[ADR 0007 — compose and moon-base](0007-compose-and-moon-base.md#workspace-shell-vs-project-services)
and the per-folder lifecycle covered in
[`specs/containers.md`](../containers.md#per-folder-compose-projects).

## Context

Project compose files we discover in bound folders almost always
ship with `restart: always` (moon-landing's `docker-compose.yml`
is the canonical example; infra repos do the same). That policy
is correct for the **project's** own developers running its
compose all day long on a workstation that reboots occasionally —
the daemon respawns mongo / redis / gitaly after host reboots so
they don't have to.

It's wrong when **moon-ide** is the lifecycle owner:

- On quit, `stop_all` snapshots which per-folder projects were
  `Running` into `session.json`'s `compose_auto_resume` map and
  then issues `docker compose stop` against each one (see
  [test plan 0015](../test-plans/0015-shutdown-stop-and-auto-resume.md)).
  With `restart: always` baked in, the daemon respawns the
  containers a few seconds later, the snapshot lies, and the user
  sees ghost services they didn't ask for. The "Stop services"
  folder-bar button has the same problem.
- More generally, moon-ide presents per-folder compose as a thing
  the user explicitly starts and stops from the IDE. A restart
  policy that contradicts the user's stop is a UX bug.

The user's compose file is **not ours to edit** — it's the
project's own source of truth, used by the project's own
developers and CI. Fixing it upstream (e.g. patching
moon-landing's compose to `restart: "no"`) doesn't scale: every
new bound folder repeats the negotiation, and the upstream policy
is correct for everyone who isn't using moon-ide.

## Decision

For per-folder compose projects only, moon-ide writes a tiny
**restart-policy override file** and layers it on top of the
user's compose via compose's native `-f` merge:

```
docker compose
    -f <folder>/docker-compose.yml
    -f <state-dir>/project-overrides/<folder-slug>.yaml
    -p moon-ws-<id>-<slug>
    <subcommand>...
```

The override looks like:

```yaml
# Generated by moon-ide. Do not edit.
services:
  mongo: { restart: 'no' }
  redis: { restart: 'no' }
  gitaly: { restart: 'no' }
  # … one entry per service the user's compose declares
```

Compose merges later `-f` files key-by-key, so the override's
`restart: "no"` replaces the user's `restart: always` while
everything else in the base file stays untouched. The user's
compose file on disk is **not** modified — a `git status` in
moon-landing stays clean.

Service names are discovered at runtime via
`docker compose -f <user-file> config --services`, which is the
exact resolver compose itself runs at `up` time. Hand-parsing
YAML to enumerate services would have to handle `extends:`,
`include:`, anchors, env interpolation, and `COMPOSE_FILE`
merging — the same set of footguns we'd have to fix every time
upstream compose adds a feature. The runtime round-trip is
~100 ms against the local daemon and runs once per mutating
command (`up`, `rebuild`, `start_service`, `restart_service`).

The override is regenerated on every `docker compose` invocation
the per-folder runner makes — adding a new service to the user's
compose and clicking "Start services" Just Works without a stale
cache.

Implementation lives in
[`crates/moon-container/src/restart_override.rs`](../../crates/moon-container/src/restart_override.rs).
[`ProjectCompose::for_folder`](../../crates/moon-container/src/project_compose.rs)
gains a `state_dir` parameter so it can resolve
`<state-dir>/project-overrides/<slug>.yaml` without a separate
config lookup.

## Why only per-folder, not the workspace shell

The workspace shell's compose (`<state-dir>/compose.yaml`) is
**generated by moon-ide** — see [ADR 0007](0007-compose-and-moon-base.md#workspace-shell-vs-project-services).
The `dev` service never gets a `restart:` key in the first
place, so there's nothing to override. The new merge codepath
(`run_docker_compose_with_overrides`) is shared between the
workspace shell and the per-folder runner; the shell just passes
an empty override list.

## Why not patch the user's compose directly

`docker compose` already has the `-f base -f override` merge
contract baked in. Using it stays within compose's documented
behaviour, doesn't require any disk writes inside the user's
repo, and survives the user editing their compose file by hand
(we regenerate the override on the next invocation). Writing the
override into our own state directory keeps "moon-ide-owned"
files separate from "user-owned" files on disk, which is the
same boundary [ADR 0007](0007-compose-and-moon-base.md) draws
for the workspace shell's `compose.yaml`.

## Known limits

- The override file enumerates the **current** service list.
  Between two `docker compose` invocations the user could add a
  new service to their compose and the override would still
  reference the old set until the next invocation regenerates
  it. That's fine — every mutating command regenerates first.
- If `docker compose config --services` itself fails (broken
  compose file, daemon down), the per-folder lifecycle command
  errors before any containers move. That's the desired failure
  mode: the user sees the _real_ problem instead of a confusing
  later error from `up`.
- The override only neutralises `restart:`. If a future need
  arises to neutralise _other_ policies (e.g.
  `deploy.restart_policy:`, `healthcheck:` retries), the same
  module is the right place to grow.
